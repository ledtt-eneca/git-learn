# ЧАСТЬ 1. Основы Git — объекты, индекс, refs, HEAD

Привет всем. Сегодня поговорим про git — но не в формате "вот команды, запомните". Поговорим о том, как он устроен внутри, потому что когда понимаешь модель — команды перестают быть магией. 

Git устроен внутри проще, чем кажется. Git — это коллекция объектов, связанных ссылками друг на друга. Каждый объект — файл в директории `.git/objects/`. И у каждого объекта есть имя — SHA-1 хеш его содержимого, 40 символов.

SHA-1 (Secure Hash Algorithm 1) — это криптографическая хеш-функция, разработанная АНБ США, которая преобразует входные данные любого размера в уникальную 160-битную (20-байтовую) строку, обычно отображаемую как 40-значное шестнадцатеричное число

1. **одинаковое содержимое всегда даёт одинаковый хеш**
2. **разное содержимое даёт разный хеш** (за редкими исключениями — "коллизиями")
3. **SHA-1 вычисляется мгновенно**

Именно это делает git надёжным: изменился хоть один байт — хеш другой. А если содержимое двух файлов одинаковое — хеш один и тот же, и данные не дублируются.

Вот как выглядит база объектов изнутри:

```
.git/objects/a4/b7fce097055c3cbd6879db9625f9a3890cc409
.git/objects/8c/3c7fbcd903744b20fd7567a1fcefa99133b5bc
.git/objects/e9/65047ad7c57865823c7d992b1d046ea66edf78
```

Первые два символа хеша становятся именем поддиректории. Такой прием ускоряет поиск объекта в 256 раз.

Алгоритм записи нового объекта следующий:
1) Git собирает структуру, данных, которая включает:
[тип объекта] [размер содержимого в байтах]\0[содержимое] 
2) Получает SHA-1
3) Сжимает содержимое полученной структуры данных 
4) Записывает на диск по правилу формирования путей

Всего в git 4 типа объектов: **blob**, **tree**, **commit**, **tag**.
Разберём каждый.


**Blob** (Binary Large Object) — хранит только содержимое файлов репозитория.

Если добавить в git файл `tutorial.txt` с содержимым `"Hello, World!"` — blob будет содержать строчку `"Hello, World!"` и ни слова о `tutorial.txt`. 

Практическое следствие: если положить в git десять одинаковых файлов по 6 мегабайт — в базе будет занято только 6 мегабайт, а не 60. Одинаковое содержимое → одинаковый хеш → один объект. Git не дублирует данные.

Blob ни на что не ссылается — он только содержит данные. Все остальные типы объектов — это по сути ссылки: либо на blob-ы, либо на другие объекты. 


**Объект tree** хранит имена файлов и структуру папок. Каждое дерево — снимок одной директории. Каждая запись в дереве содержит четыре поля:

- **mode** — права доступа в UNIX-стиле: `100644` обычный файл, `100755` исполняемый, `120000` симлинк, `040000` директория
- **type** — `blob` (файл) или `tree` (вложенная папка)
- **SHA** — ссылка на соответствующий объект
- **name** — имя файла или папки в этой директории

На один blob могут ссылаться сразу несколько tree.

Поддиректория — это просто ещё один tree-объект, на который ссылается родительский tree. Всё дерево папок проекта — граф вложенных tree-объектов с blob-ами на листьях.

Изменился один файл глубоко в поддиректории? Git пересоздаёт только blob для этого файла и все tree по пути от него до корня. 

И важный момент: хеш tree идентифицирует — все файлы, все поддиректории. Одинаковый хеш двух tree = идентичное содержимое. Git сравнивает два состояния одним числом, без чтения файлов.


```
commit abc123
    tree:      → ccc333  (корневой tree — полный снимок проекта)
    parent:    → xyz789  (родительский коммит)
    author:    Vasya <vasya@example.com>
    committer: Vasya <vasya@example.com>
    date:      2024-03-19 15:42
    message:   "feat: add hello"
```

**Commit** — объект, который хранит историю. Он содержит:
- ссылку на корневой tree — полный снимок проекта на момент коммита
- ссылки на родительские коммиты
- **author** — кто написал код, **committer** — кто применил коммит к репозиторию
- дату и сообщение

У самого первого коммита нет родителей. У большинства коммитов один родитель — история линейна. Если у одного коммита два потомка — история разошлась на ветви. Если у коммита два родителя — ветви сошлись, это merge-коммит.

Обычно author и committer совпадают. Они расходятся когда один человек написал патч, а другой его применил — например при `cherry-pick` или работе через email-патчи.

Хеш коммита вычисляется из всего содержимого — tree, parent, author, date, message. Изменить что-либо = получить другой хеш = новый объект. Именно поэтому `rebase` и `commit --amend` не редактируют старые коммиты — они создают новые. Старые остаются в базе нетронутыми (и достижимы через `reflog`).

---

Четвёртый тип объектов — **tag**. Теги бывают двух видов.

**Лёгкий тег** — просто указатель, ссылка на коммит. Создаётся `git tag v1.0`. Никакого объекта в `.git/objects/` не создаётся — только запись в `.git/refs/tags/`.

**Аннотированный тег** — полноценный объект в базе. Создаётся `git tag -a v1.0 -m "Release v1.0"`. Он хранит ссылку на коммит, имя тегирующего, дату и сообщение. Может быть подписан GPG.

## Три области — рабочая папка, индекс, репозиторий

В git можно выделить три области. 
**Рабочая папка** — обычные файлы на диске, вы их редактируете. 
**Репозиторий** — объекты в `.git/objects/`, неизменяемые. 
**Индекс** — посередине.

Индекс — это файл `.git/index`. В нём хранится список всех отслеживаемых файлов: для каждого файла — mode, SHA его staged blob, и путь. Фактически это "черновик следующего tree".

`git add` создает blob и обновляет его имя в индексе. 
`git commit` берёт индекс, строит из него дерево tree-объектов, создаёт commit-объект, двигает HEAD.


`git status` сравнивает сразу два места:
- **HEAD vs индекс** — то, что вы уже сделали `git add` (Changes to be committed)
- **индекс vs рабочая папка** — изменения которые вы ещё не `add`-нули (Changes not staged)

`git diff` — показывает разницу между рабочей папкой и индексом (что не в staged).  
`git diff --staged` — между индексом и HEAD (что пойдёт в следующий коммит).

## Ветки и HEAD

**На слайде:**
```
.git/refs/heads/main     → содержит: "abc123"
.git/refs/heads/feature  → содержит: "def456"
.git/HEAD                → содержит: "ref: refs/heads/main"

Ветка = текстовый файл с хешем последнего коммита
HEAD = указатель на текущую ветку
```

Ветка — это буквально текстовый файл. Откройте `.git/refs/heads/main` — там будет 40-символьный хеш. Всё. Создать ветку — создать такой файл. Удалить ветку — удалить файл. Вот почему ветки в git такие дешёвые.

HEAD — специальный файл, который указывает, на какой ветке вы сейчас. Обычно HEAD указывает не на коммит напрямую, а на ветку: HEAD → main → abc123.

Когда вы делаете коммит:
1. Создаётся новый commit-объект
2. Файл ветки (main) обновляется — туда записывается хеш нового коммита
3. HEAD по-прежнему указывает на main, а main теперь смотрит на новый коммит

Ветка — это просто движущийся указатель на последний коммит. Она "идёт вперёд" с каждым коммитом. История при этом не меняется — старые объекты никуда не деваются, они просто перестают быть "кончиком ветки".

---

## Как обращаться к коммитам — ~ и ^


Во многих командах нужно указать "какой коммит". Можно вставить SHA-1 хеш или использовать HEAD.

Тильда `~N` — это "N шагов назад по истории". Обычно именно это и нужно.

```
C5 ← C4 ← C3 ← C2 ← C1
↑
HEAD (= C5)

HEAD~1 = C4   (на один назад)
HEAD~2 = C3   (на два назад)
HEAD~3 = C2   (на три назад)
```

Как это выглядит:
- `git reset HEAD~2` — откатиться на два коммита
- `git rebase -i HEAD~4` — взять в редактор последние 4 коммита
- `git diff HEAD~1` — посмотреть что изменилось в последнем коммите

Работает не только с HEAD. Можно писать `main~3`, `feature~1` — от любой ссылки.

---

### Каретка ^ 

`^` — это выбор родителя. У обычного коммита один родитель: `HEAD^` = `HEAD~1`, это одно и то же. Но у **merge-коммита два родителя** — и вот тут `^` становится нужным.

```
           ┌─── B2 ← B1
           ↓
MERGE ←────┤              MERGE^1 = A3  (ветка, в которую мержили)
           ↑              MERGE^2 = B2  (ветка, которую мержили)
           └─── A3 ← A2
```

```
MERGE^1  — первый родитель  (ветка, на которой стояли при git merge)
MERGE^2  — второй родитель  (ветка, которую передали в git merge)
MERGE~1  — то же что MERGE^1 (тильда всегда идёт по первому родителю)
```

На практике `^2` нужен редко — в основном когда разбираете запутанную историю с merge-коммитами или пишете скрипты. Но знать об этом полезно, потому что иначе непонятно почему `~` и `^` иногда дают разные результаты.

Можно комбинировать: `HEAD~2^2` — два шага назад, затем второй родитель.

---

### Полная шпаргалка по адресации

**На слайде:**
```
HEAD          — текущий коммит
HEAD~1 = HEAD^ — предыдущий коммит
HEAD~N        — N коммитов назад
HEAD^2        — второй родитель merge-коммита
<hash>        — конкретный коммит по SHA-1 (можно первые 7 символов)
main~3        — три коммита назад от кончика ветки main
v1.0~1        — коммит перед тегом v1.0
```

# ЧАСТЬ 2. Ветвление — merge и rebase

Мы разобрали как git хранит данные — объекты, индекс, ссылки. Теперь посмотрим как работает ветвление. Всё что будет дальше — это просто манипуляции с указателями, которые мы только что обсудили.

Когда создаёте ветку — git просто создаёт новый файл с тем же хешем. Никакого копирования данных. Обе ветки смотрят на один и тот же коммит. 


---

## Merge

Merge создаёт новый коммит с двумя родителями — по одному от каждой ветки. История сохраняется полностью. Можно увидеть, что было в feature, что было в main, когда они соединились.

Есть особый случай — **fast-forward merge**. Если main не ушёл вперёд после создания feature — то есть feature просто добавила коммиты поверх — тогда git не создаёт merge-коммит. Он просто двигает указатель main вперёд. Линейная история.

---

## Rebase

Rebase — это переписывание истории. Он берёт ваши коммиты  и пересоздаёт их заново, но уже поверх другого основания. Получаются новые коммиты — с тем же содержимым изменений, но другими хешами, другими родителями, другими датами создания.

Главное правило, которое нужно запомнить: **никогда не делайте rebase коммитов, которые уже есть у других людей**. Это поломает им историю. Rebase — только для локальных веток, которые ещё не пушили. Или для своей personal feature-ветки, если все знают, что вы её перепишете.

---

# ЧАСТЬ 3. Удалённые репозитории

Ветвление разобрали. Теперь — как всё это работает когда репозиториев больше одного.

---

## [СЛАЙД: Как устроена работа с remote]

Удалённый репозиторий — такой же git-репозиторий, только на другой машине. Те же объекты, те же refs. Принципиальных отличий в устройстве нет.

`origin` — просто имя-алиас для URL. `origin` — соглашение, можно назвать иначе, можно иметь несколько remote'ов.

`origin/main` — это **ваша локальная** read-only копия того, в каком состоянии `main` был на сервере, когда вы последний раз синхронизировались. Вы на неё не коммитите — только смотрите.

---

`git fetch` — всегда безопасно. Скачивает новые объекты, обновляет `origin/main` и другие remote-tracking ветки. Ваши локальные ветки не меняет. Хорошая привычка — делать перед началом работы.

`git pull` = `git fetch` + `git merge`. Сначала скачивает, потом сливает в текущую ветку. Есть подводный камень — об этом в следующем слайде.

`git push -u origin feature/auth` — первый push новой ветки. Флаг `-u` пушит ветку **и** настраивает tracking: после этого просто `git push` и `git pull` без аргументов.

## [СЛАЙД: Три типичных сценария]

**Сценарий 1: git pull создаёт мусорный merge-коммит**

> Вы закоммитили локально. Коллега успел запушить. `git pull` вклинивает `Merge branch 'feature/auth' of github.com/...`.

Ветки разошлись — git создаёт merge-коммит чтобы их соединить. Решение: `git pull --rebase` — ваши коммиты перекладываются поверх того, что пришло с сервера. История линейная.

```bash
git pull --rebase

# Конфликт при rebase:
git add <файл>
git rebase --continue
git rebase --abort          # или отменить — вернётся к состоянию до pull

# Настроить глобально раз и навсегда:
git config --global pull.rebase true
```

**Сценарий 2: push отклонён после rebase**

После `rebase -i` хеши изменились — сервер отклонит обычный push, история "расходится". Нужен force push.

`--force-with-lease` безопаснее `--force`: откажет если кто-то ещё запушил в эту ветку пока вы работали.

```bash
git push --force-with-lease
# git push --force  ← опасно, сотрёт чужие коммиты
```

**Сценарий 3: работа с веткой коллеги**

`origin/feature/auth` — read-only, коммитить нельзя. Нужна локальная ветка:

```bash
git checkout feature/auth               # git сам создаст из origin/feature/auth
# или явно:
git checkout -b feature/auth origin/feature/auth

git branch -a    # все ветки: локальные + remote-tracking
git branch -r    # только remote-tracking
```

# ЧАСТЬ 4. Команды и сценарии

---

## git stash

`git stash` — временный карман. Прячете незакоммиченные изменения, переключаетесь куда нужно, делаете что нужно, возвращаетесь и достаёте обратно.

`pop` vs `apply` — ключевое отличие: `pop` достаёт и удаляет запись из стека. `apply` достаёт, но оставляет. Удобно если хотите применить одни и те же изменения в нескольких ветках — применили в одной, потом в другой, потом явно `drop`.

---

## git cherry-pick

**На слайде:**
```
git cherry-pick <SHA>          — взять один коммит
git cherry-pick <SHA1> <SHA2>  — взять несколько
git cherry-pick <SHA1>..<SHA2> — диапазон (SHA1 не включается)
```

cherry-pick — "хочу вот тот конкретный коммит, но в эту ветку". Git берёт изменения из коммита и применяет к текущей ветке. Создаётся новый коммит с тем же содержимым изменений, но другим хешем.

Это копирование, не перемещение. Оригинал остаётся на месте.

Почему нельзя `HEAD~N`? Потому что `HEAD` указывает на текущую ветку. Нужный коммит живёт в `feature/auth` — адресовать его можно только по SHA.

```
git log feature/auth --oneline   ← смотрим SHA нужного коммита
git checkout main
git cherry-pick <SHA>             ← копируем
```

## git revert

**На слайде:**
```
git revert <SHA>        — отменить конкретный коммит (безопасно)
git revert HEAD         — отменить последний коммит
git revert HEAD~1       — отменить предпоследний
git revert <SHA> --no-edit  — без открытия редактора
```

### Сценарий

> Коммит с багом ушёл в `main`. Коллеги уже стянули его к себе. Нельзя просто удалить из истории — нужно отменить изменения, но сохранить историю.

**Говорим:**

revert — безопасная отмена для shared веток. Создаёт новый коммит, который применяет обратные изменения. Оригинальный коммит остаётся в истории.

`revert HEAD` vs `revert <SHA>` — не одно и то же в общем случае. `HEAD` — это "прямо сейчас", работает если плохой коммит последний. Если нет — используйте SHA. Так безопаснее и явнее.

---

## git reset

*Слайд: таблица трёх режимов + схема движения HEAD*

**На слайде:**
```
git reset [--soft | --mixed | --hard] <commit>

Режим     Рабочая папка   Индекс    HEAD
──────────────────────────────────────────
--soft         ✓             ✓       ←
--mixed        ✓             ←       ←    (по умолчанию)
--hard         ←             ←       ←

✓ = не тронут    ← = сброшен к состоянию коммита
```

**Говорим:**

reset — это перемещение HEAD (и ветки) на другой коммит. Три режима отличаются тем, что происходит с индексом и рабочей папкой.

```
До:    main → C4 ← C3 ← C2 ← C1
                ↑ HEAD

После git reset HEAD~2:
       main → C2 ← C1
               ↑ HEAD

C3, C4 физически никуда не делись — живут в .git/objects/
Но ветка их больше не видит. Достать — через git reflog.
```

### --soft: склеить мусорные коммиты

**Сценарий:** работали над фичей, коммитили по ходу — "начал", "добавил", "fix typo", "ещё fix". Перед PR хочется один аккуратный коммит.

`--soft` перемещает HEAD назад, все изменения остаются в индексе (staged). Как будто коммитов не было — всё готово к одному чистому коммиту.

```
git reset --soft HEAD~3    ← три коммита назад
git reset --soft <SHA>     ← или явно по SHA нужного коммита
git commit -m "feat: готово"
```

*→ демо: demo-commands.sh, блок `reset --soft`*

### --mixed: убрать из staged

**Сценарий:** сделали `git add .` — добавили всё подряд включая отладочный файл. Нужно убрать лишнее из индекса, не теряя изменений.

`--mixed` (default) сбрасывает индекс до состояния коммита. Рабочая папка не трогается.

```
git reset HEAD           ← весь индекс
git reset HEAD debug.py  ← только один файл
git reset                ← то же самое, HEAD подразумевается
```

*→ демо: demo-commands.sh, блок `reset --mixed`*

### --hard: "откатить всё"

**Сценарий:** запустили AI-агент — он прошёлся по файлам, что-то понарефакторил. Результат не нравится, хочется вернуть всё как было одной командой.

`--hard` сбрасывает и индекс, и рабочую папку. Незакоммиченные изменения теряются.

```
git reset --hard HEAD      ← откат к последнему коммиту
git reset --hard HEAD~2    ← откат на 2 коммита назад
git reset --hard <SHA>     ← откат к конкретному коммиту
```

**Важно:** `--hard` на shared ветке (main, develop) — плохая идея, если коллеги уже стянули эти коммиты. Для shared веток — `revert`. `reset` — только для локальной истории.

*→ демо: demo-commands.sh, блок `reset --hard`*

## git rebase -i

*Слайд: до/после — 4 коммита слева, 2 чистых справа; список операций*

**На слайде:**
```
git rebase -i HEAD~N          — редактировать последние N коммитов
git rebase -i <SHA-предка>    — всё от предка до HEAD
git rebase -i $(git merge-base HEAD main)  — от точки ветвления

Операции в редакторе:
  pick    — оставить
  reword  — переименовать
  squash  — склеить (сохранить сообщение)
  fixup   — склеить (отбросить сообщение)
  drop    — удалить
```

**Говорим:**

Интерактивный rebase — редактор истории. Git открывает список ваших коммитов, вы указываете что с каждым сделать. Старые хеши уходят, появляются новые.

Коммиты в редакторе идут **старые сверху, новые снизу** — обратно `git log`.

```
git rebase -i HEAD~4
# В редакторе:
#   pick abc1 feat: login stub
#   fixup abc2 fix: return true   ← склеить с abc1
#   pick abc3 feat: logout stub
#   drop abc4 WIP                 ← удалить
```

Если что-то пошло не так:
```
git rebase --abort              ← отменить, вернуться назад
git reset --hard ORIG_HEAD      ← после завершения если передумали
```

После rebase если уже пушили — нужен `git push --force-with-lease`.

*→ демо: demo-commands.sh, блок `git rebase -i`*
